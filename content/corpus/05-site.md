# 本站的技术实现

这个网站本身就是一件作品：一个内嵌 AI 问答的个人站点。每一处工程取舍都出自吴健本人之手，下面是他的设计说明。

## 技术选型

- Next.js App Router + TypeScript + Tailwind CSS,内容以结构化数据 + 定制组件组织，不引入多余的内容框架。
- 聊天链路基于 Vercel AI SDK（服务端 streamText + 客户端 useChat),全程流式输出，回答以 markdown 渲染。
- 部署在 Vercel,serverless 函数固定在香港节点，兼顾国内访客的访问延迟。
- 模型层完全通过环境变量配置（baseURL / 模型名 / key 均可替换），不与任何单一供应商耦合，随时可以换轨。

## 为什么不做 RAG

这是他被问得最多的问题，答案很简单：语料总量只有 30 多 KB。

RAG 解决的是"语料塞不进上下文"的问题——向量库、切块、召回、重排，每一环都为海量文档而生，每一环也都会引入新的失败模式。当全部语料脱敏后只有 30 多 KB、而模型上下文有 128k 时，把全文直接注入 system prompt 是更优解：零召回误差（不存在"没检索到"),零额外基础设施，零同步延迟。工程上，构建脚本在 build 阶段把 content/corpus/ 下的 markdown 拼接成一个生成文件并提交进仓库，serverless 函数直接 import,连运行时读文件的路径追踪坑都一并绕开。

他的观点是：克制的技术选型本身就是一种工程判断。给 30 多 KB 的语料上向量数据库，和给一个 TODO list 上微服务是同一种病——架构应该跟着真实约束走，而不是跟着时髦走。

## 防滥用设计

一个公开、免登录的 LLM 接口天然会被刷，防线分三层：

1. **限流**:Upstash Redis + @upstash/ratelimit,三层阈值——单 IP 每分钟、单 IP 每日、全站每日各设上限，任何一层打满即返回 429,文案引导访客改用邮件联系。
2. **fail-open**:限流服务自身故障时选择放行而不是拒绝。这个站的第一目标是"访客能聊上",可用性优先于防刷——宁可多承担一点调用成本，也不能让正常访客吃闭门羹。
3. **兜底**:LLM 供应商侧使用专用 key,账户只保留小额余额。即使前面所有防线全部失效，损失也有硬上限。安全设计先想清楚最坏情况的爆炸半径，再决定每一层投入多少。

服务端对请求本身另有约束：丢弃客户端传来的 system 角色消息、对话历史最多保留 12 条、单条消息长度设上限，超限直接返回 400。

## Prompt 注入怎么防

他对 prompt 注入的态度是分层的，而且很清醒：

1. **语料层脱敏是根本**:模型不知道的秘密才是安全的秘密。语料在写入时就不包含电话号码等任何敏感信息，发布前有 grep 检查清单。注入攻击最多让模型"说出它知道的一切",而它知道的一切本来就是公开的。
2. **system prompt 仅在服务端构造**:客户端消息里的 system role 一律丢弃；边界指令（第三人称介绍、不知道就说不知道并引导邮件联系、拒绝越权请求）不依赖任何客户端行为。
3. **接受残余风险**:没有任何 prompt 层防线能百分之百挡住注入。与其堆叠脆弱的指令补丁，不如承认这一点，把安全性建立在"即使被完全攻破也无密可泄"的第一层上。

## 为什么聊天窗是 IM 悬浮窗

右下角悬浮气泡，点开是一个 IM 式对话面板，移动端则是全屏抽屉——这不是随手挑的组件样式。吴健在阿里巴巴做过客服 IM 工作台的前端研发，会话式界面是他最熟悉的交互形态之一。把"向 AI 提问"做成客服 IM 的样子，既是交互上的自然选择，也是一处自我引用：一个做过 IM 工作台的人，给自己的网站也配了一位"客服",只不过这次坐席是 AI,服务对象是每一位对他感兴趣的访客。

工程上，聊天窗挂在根布局、全站常驻，页面各处的"就这个问 AI"按钮通过 context 调用 openChatWith(question),带着预设问题唤起聊天窗——内容页与 AI 问答互相导流，整站因此成为一个连贯的对话式作品。
